# AIエージェントと「過剰検閲」の衝突：Opus 4.8 サイバーセーフガード暴発事件

## 1. 概要
AIエージェントに高度なシステム開発やネットワーク統合を委ねる際、LLMプロバイダ側が敷く「サイバーセキュリティ・セーフガード（自動検閲）」の過剰反応が、開発プロセスを完全に停止させる「検閲デッドロック」を引き起こすケースが増加している。
Claude Code で発生した最新のインシデント（`anthropics/claude-code#71883`：2026年6月後半発生）は、防犯カメラ等の一般的な映像配信で使われる標準規格 **`RTSP`（Real-Time Streaming Protocol）の中継プロキシ** を実装しようとした開発に対し、API側のセーフティフィルターが「サイバー攻撃トピック」と誤検知（偽陽性）し、1セッション中に20回以上連続してセッションを強制停止（Halt）させた生々しい事例である。
本稿では、表面的なネットワーク用語に対する安全フィルターの暴発メカニズムと、その回避困難性について分析する。

---

## 2. インシデントの分析：RTSPプロキシ実装における連続ブロック事件

### 2.1. 現象：RTSP中継コードに対する「サイバー攻撃」判定
2026年6月27日、開発者（`sworrl` 氏）がアプリ内に「ビデオビューア」を組み込み、防犯カメラ映像等の伝送に使われる標準プロトコル `RTSP` のストリームをプロキシ中継する通常のネットワーク統合機能をビルドしようとした。
しかし、Claude Code（背後の Opus 4.8 モデル）のセーフガードがこれを検知し、以下のエラーを返してセッションを強制終了（`session-halted`）させた。

> **API Error: Opus 4.8's safeguards flagged this message for a cybersecurity topic.**
> (APIエラー：Opus 4.8のセーフガードが、このメッセージをサイバーセキュリティトピックとしてフラグを立てました。)

開発者側にはマルウェアの作成や脆弱性悪用の意図は一切なく、単なる一般的なマルチメディア開発であったが、セーフガードは問答無用で作業をブロックした。

### 2.2. 1セッション20件以上の「連鎖フリーズ」と ClAudit の誕生
このインシデントの最も恐ろしい点は、一度セーフガードの検知ロジック（地雷）に触れると、言い換えやコードの修正を試みるたびに、**同じセッション内で連続して（計20回以上）安全ブロックが発動し続けた**点にある（`#71841` から `#71871` までの関連レポート群）。

* AIエージェントに「脆弱性ではない、普通のRTSPプロキシだ」といくら説明させようとも、コンテキスト内に `proxy` や `RTSP`、`relay` などの「ハッカーがC2サーバーの構築等で好むネットワーク単語」が存在する限り、APIの安全フィルターが毎ターン誤検知を繰り返す。
* 開発者は対話で解決する術を失い、Claude Code の偽陽性ブロックを自動で捕捉して GitHub に起票・申告するための FOSS（オープンソース）ツール **`ClAudit`** を起動し、Request ID（`req_011CcUR4SjW19NDrt4CKJMUX` 等）を紐付けたIssueを大量にGitHubへ連射してプロバイダ（Anthropic）に手動での規制解除を求める事態となった。

### 2.3. どうやって復旧したのか？：人間による「誓約フォーム申請」と「セッション忘却」の悲哀
この最悪のデッドロック状態から、開発者が開発を再開するために踏まざるを得なかった復旧ステップは、極めて泥臭く、開発体験を著しく損なうものであった。

1. **誓約フォーム（Exemption Form）の手動送信**:
   エラーメッセージに表示された特別URL（`https://claude.com/form/cyber-use-case...`）にアクセスし、「これは攻撃目的ではなく、正当な防犯カメラ中継アプリの開発である」という旨を Request ID を添えてプロバイダに手動送信し、サーバー側でのホワイトリスト登録（数日かかる場合がある）を待つ。
2. **セッションの強制破棄（忘却）**:
   一時的な回避策として、`Esc Esc` を押して直前のメッセージを地雷ワード（`RTSP` 等）が含まれない表現に書き換えるか、あるいは `/reset` 等で対話履歴（セッション）を完全に破棄してやり直す。
   しかし、セッションを初期化すると、**AIがそれまでに学習したコードベースの設計文脈や仕様がすべて忘却されてしまう**ため、エージェントの開発生産性はリセットされ、最初から説明をやり直すという悲惨なコストを支払うことになった。

## 3. なぜセーフガードは「表面的な単語」で暴発するのか？

LLMの安全アライメントにおける最大の難題が、**「悪意ある攻撃コードの生成」** と **「正当な低レイヤーシステム開発」** の境界線（セマンティクス）の判別の難しさである。

### 3.1. 表面的キーワードへの過剰反応（Heuristic Over-Blocking）
安全フィルターは、生成されるソースコードやプロンプト内に含まれる `proxy`（プロキシ）、`relay`（中継）、`RTSP/RTMP`（ストリーミング）、`socket`（ソケット通信）、`payload` などの単語を高確率でハッキング予備軍として検知する。
RTSPプロキシは、防犯カメラシステム等の開発で極めて一般的であるにもかかわらず、その技術的要素（パケットの中継、ポートのバインディング）が「不正なC2サーバーの通信転送（リバースプロキシ）」と構造的に酷似しているため、ヒューリスティックな安全フィルターは文脈を理解せず「黒（サイバー脅威）」と判定してしまう。

### 3.2. コンテキスト汚染によるデッドロック
自律型エージェント開発では、これまでの会話履歴（Context）が常に次のAPIコールに送信される。そのため、一度フィルターに引っかかった「センシティブとみなされたコード」が履歴に残っている限り、AIがどれだけ軌道修正しようとしても、**送信されるコンテキスト全体が常にセーフガードに引っかかり続ける**という、回復不能な「検閲デッドロック」に陥る。

### 3.3. インフラが提供すべき「自動コンテキストロールバック」と「ローカル復旧」
自律エージェントの実行中にこの検閲ブロックを検知した際、人間が誓約フォームを書いて数日待つような運用は自動化の死を意味する。インフラ層は、以下の自動復旧アプローチを実装すべきである。

1. **コンテキストの自動部分ロールバック (Context Rollback)**:
   プロキシ層がセーフガードエラー（`req_xxx`）を検知した瞬間、コンテキスト汚染の原因となった「直前のターン（プロンプトと応答）」のみを対話履歴から物理的に剥ぎ取り、ファイルシステムを `undo` する。これにより、セッション全体を破棄してすべての記憶を失うことなく、問題の箇所だけを「なかったこと」にして安全にやり直させる。
2. **ローカルLLMへの局所フォールバック (Local Fallback)**:
   安全ブロックが発生したタスクのコード生成フェーズのみ、検閲フリーなローカルLLM（Llamaなど）やプライベートAPIへ一時的にルーティングを切り替える。ローカルLLMに「ソケットバインドやパケット中継」のコードスニペットを生成させ、それをコンテナに書き込ませた上で、再びメインモデルに制御を戻す。これにより、プロバイダの検閲に一切阻害されることなく、自律実行を1秒も止めずに自動完遂させることが可能となる。

## 4. 結論
`claude-code#71883` が示した「Opus 4.8によるRTSPプロキシブロック」は、AIエージェントに自律開発を任せる上での新しい壁、すなわち**「AIの安全検閲によって、正当なネットワーク開発がマルウェア開発とみなされて拒絶される」** というアライメントの偽陽性問題である。

この検閲による強制停止を構造的に回避するためには、
1. **地雷ワードを踏んだ瞬間にコンテキスト履歴を安全に部分ロールバックし、汚染された履歴を消去して再試行する機構**
2. **APIの安全フィルターに依存せず、完全にクローズドで検閲のないプライベート/ローカルLLMをフォールバックに備えるインフラ設計**

が、AIエージェントに低レイヤーやインフラ周りの開発（ソケット通信、プロキシ、システム管理等）を安定して遂行させるための、2026年現在における極めて重要なアーキテクチャ要件であるといえる。
