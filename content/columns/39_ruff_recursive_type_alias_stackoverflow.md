# 再帰の奈落：Ruffの型推論エンジンをスタックオーバーフローで即死させた Python 3.12 循環型エイリアスの罠

## 1. 概要
静的解析ツールやコンパイラ（Ruffの次世代型型チェッカーなど）は、ユーザーが記述した不完全、あるいは不正なプログラムコードであっても、決してクラッシュ（パニック）せずに安全にエラーを出力しなければならない。
しかし、言語仕様に「再帰的な定義」が加わると、型解析器の内部ロジックにおいて無限再帰（Infinite Recursion）の罠が口を開ける。
本稿では、Ruffの型解析器（`ty` クレート）で発生した、Python 3.12の再帰的型エイリアスによるスタックオーバーフロークラッシュ（`astral-sh/ruff#26690`）を事例に、再帰的静的解析における循環参照検知（Cycle Guard）の重要性について分析する。

---

## 2. インシデント of 分析：解析対象のコードによって「即死」させられる解析器

### 2.1. 現象：型チェック実行時の突然のスタックオーバーフロークラッシュ
Ruffで特定のPythonファイルをチェックしようとした際、何のエラーハンドリングも通らず、プロセスが突然スタックオーバーフロー（Stack Overflow）を引き起こしてクラッシュするバグが報告された。

静的解析ツールにおいて、特定のコードを入力しただけでプロセスが即死するバグは、IDEの応答不能やCI/CDパイプラインの強制終了を招く極めて深刻な可用性障害（DoS状態）に該当する。

### 2.2. 原因：Python 3.12の「型エイリアス」による循環参照
このバグの原因は、Python 3.12で導入された PEP 695 の新しい型エイリアス構文（`type` ステートメント）を用いて、以下のような「お互いがお互いを参照し合う循環構造」を記述したコードをRuffに読み込ませたことにあった。

```python
type NonData = int | NonDataTail
type NonDataTail = NonData  # 循環参照！
```

Ruffの型推論エンジンは、属性アクセス時の優先順位を判定するため、対象のクラスやオブジェクトが「データデシケータ（Data Descriptor：`__set__` や `__delete__` を持つもの）」であるかを分類する処理（`Descriptor Classification`）を走らせる。

この分類器（Classification Predicate）は、対象の型が Union（合併型）やエイリアスである場合、その中身を再帰的にドリルダウンして検査する設計になっていた。
しかし、この再帰ロジックに**「一度訪れた型エイリアスを記録し、二度目の到達で再帰を打ち切る」という巡回追跡ガード（Visited Guard / Cycle Guard）が存在しなかった**。

そのため、解析器は `NonData` ➔ `NonDataTail` ➔ `NonData` という無限ループ（再帰呼び出し）に突入し、一瞬でスタックフレームを消費し尽くして OS レベルでプロセスが強制終了された。

### 2.3. 「バグを指摘するツール」がバグによって即死する自己言及的崩壊
循環参照（Circular Reference）がシステム開発において最も恐ろしいアサシン（暗殺者）と呼ばれる理由は、**「バグ（間違い）を検知して指摘するはずのツール自体が、そのバグによって破壊される」** という自己言及的な脆弱性に直結するからである。

開発者がうっかりタイプミス等で循環したエイリアスや依存関係を書いてしまうことは日常茶飯事である。
リンターや型解析器は、そうした「間違った壊れたコード」を安全にパースしてエラーをユーザーにフィードバックするのが本来の役割である。
しかし、入力データ（AST）に循環が生まれた瞬間、解析器自身が無限ループに入ってパニック死してしまっては、バグを修正するための道具としての信頼性が完全に喪失する。

また、循環参照はプロセスを即死させるだけでなく、参照カウント（Reference Counting）方式のメモリ管理において、オブジェクトが互いを参照し合うことでカウントが永久に「1」のまま残り続ける **メモリリーク** の最大の要因でもある。
言語仕様（Pythonの巡回GCや、Rust/C++の `Weak` ポインタ）が多大なランタイムコストを支払ってでもこの問題に対抗していることからも、循環参照がソフトウェアのライフサイクルにおいていかに普遍的で破壊的な脅威であるかが伺える。

## 3. 実体験：AST（抽象構文木）の走査における「循環リンク」の無限再帰クラッシュ
ファイルシステム、依存関係グラフ、あるいはAST（抽象構文木）など、「グラフ構造」を持つデータを再帰で走査する際、巡回判定（Cycle Detection）を1箇所でも忘れると、容易にシステム全体のクラッシュを招く。

* **実体験エピソード**: 
  社内の大量のマークダウンファイルを解析し、`[リンクテキスト](file.md)` のような相対リンクを辿って、ファイル間の関連図（ナレッジグラフ）を自動生成するクローラーを開発した。
  「あるファイル A.md が B.md を指し、B.md が C.md を指す」という関係を、深さ優先探索（DFS）の再帰関数で再帰的にパースし、親子の依存関係をマップ化するシンプルなコードだった。
  リリース後、特定のフォルダ群をクロールさせた瞬間、CPU使用率が100%に張り付き、最終的に `RangeError: Maximum call stack size exceeded` でシステムがクラッシュした。
  原因は、ドキュメントの編集者が、A.md の中に B.md へのリンクを書き、かつ B.md の中にも A.md へのリンクを書くという **「循環参照（自己参照リンク）」** を記述していたことだった。
  「一度訪問したノードIDを記録する Set（集合）」のチェックを、コードの最適化の際につい省略してしまったため、クローラーは A ➔ B ➔ A ➔ B と無限に巡回し、スタックを食いつぶした。
  再帰を使うロジックにおいて、「グラフには必ずループ（循環）が存在する」という前提でガードを書かないシステムは、ユーザーの入力次第でいつでも爆破できる爆弾と同じである。

## 4. 解決策と「再帰走査」における設計原則

Ruffの開発チームは、PR #26690 において、型分類器の最深部に **Cycle Guard（循環保護）** を追加した。

### 4.1. 再帰評価時の「訪問履歴（Visited）」の記録とスタック制限
型推論器がエイリアスを再帰的に解決する際、訪問中のエイリアスの識別子をスレッドローカルな状態（あるいはスタックフレーム）に記録し、すでに走査中の型が再登場した場合は、評価を即座に打ち切って `Unknown` や `Any` などのフォールバック値を返す。

* **修正の論理イメージ**:
  ```rust
  fn is_data_descriptor(
      &self, 
      ty: &Type, 
      visited: &mut HashSet<TypeId> // 循環検知用のVisitedガード
  ) -> bool {
      if !visited.insert(ty.id()) {
          return false; // すでに訪問済みの場合は再帰を打ち切る (Cycle detected)
      }
      
      // 再帰的な評価処理...
      match ty {
          Type::Union(variants) => {
              variants.iter().any(|v| self.is_data_descriptor(v, visited))
          }
          // 他の型判定...
      }
  }
  ```

## 5. 結論
言語解析器やインフラツールを設計する際、ユーザーの入力データ（コードやファイルシステム）は「常に悪意ある、あるいは壊れた循環構造を含んでいる可能性がある」という前提（ゼロトラスト・データ）で実装しなければならない。
再帰を用いた走査を行うすべてのロジックにおいて、**「循環参照の早期脱出ガード（Cycle Guard）を義務付ける」**、**「スタックの深さにハードリミットを設ける」** というセーフティネットを張ることは、可用性（DoS耐性）を守るための絶対的な設計原則である。
